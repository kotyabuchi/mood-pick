## 議論サマリー

### 合意点（初回で一致、即採用）

- C-1/A-2: Phase 4 の「おすすめ受信時」通知トリガーを Phase 5 に移動
- C-2/A-3: recommendations は専用テーブルとして確定（activity_log とは別ドメイン）
- C-6/A-5: handle バリデーション（英数字+アンダースコア、3-20文字、小文字正規化）を Phase 2 に追加

### ラリーの記録

#### C-3: マイグレーション分離コミットリスク
- **R1 Codex**: DB変更とアプリ変更の分離コミットで本番不整合リスク。アトミックデプロイを推奨
- **R1 Claude**: 個人開発 PWA で Supabase hosted + Vercel デプロイ。アトミックデプロイのコストに見合うか？ 後方互換マイグレーションルールで十分では？（聞き返し）
- **R2 Codex**: 【修正】個人開発前提なら「アトミック配備の仕組み必須」は過剰。「後方互換マイグレーション（expand/contract）を徹底」ルールで実害の大半は回避
- **結論**: 採用（修正版）— 後方互換マイグレーションルールを「全体ルール」に追記

#### C-4: RLS テスト自動化
- **R1 Codex**: RLS テストの CI 自動化が必要
- **R1 Claude**: pgTAP のセットアップコストがこの規模に見合うか？ 手動確認では不十分か？（聞き返し）
- **R2 Codex**: 【修正】pgTAP 必須は撤回。`psql` ベースで owner/other/anon の許可/拒否を数ケース確認する最小限のスモークテストを提案
- **結論**: 採用（修正版）— psql ベースのスモークテスト方針を追記

#### C-5: activity_log トリガー冪等性
- **R1 Codex**: トリガーに冪等性/重複防止がない
- **R1 Claude**: PostgreSQL トリガーは1トランザクション内で1回のみ発火。具体的な重複シナリオは？（聞き返し）
- **R2 Codex**: 【維持】再送・二重クリック・同値UPDATE・状態往復で同種イベントが複数 INSERT される。`WHEN OLD.status IS DISTINCT FROM NEW.status` が有効
- **R2 Claude**: 状態往復（want→watching→want）での重複は確かに起こりうる。WHEN 条件は低コストで有効
- **結論**: 採用 — Phase 3 のトリガーに `WHEN OLD.status IS DISTINCT FROM NEW.status` を追記

#### C-7: ロールバック/バックフィル手順
- **R1 Codex**: ロールバックとバックフィル手順が不足
- **R1 Claude**: 新テーブルは空から始まるのでバックフィル不要では？（聞き返し）
- **R2 Codex**: 【修正】初期バックフィルは撤回。DOWN マイグレーションのコメント記載を推奨
- **結論**: 採用（修正版）— DOWN マイグレーションコメント方針を追記

#### A-1: profiles SELECT RLS の開放
- **R1 Claude**: 現在の profiles RLS は自分のみ SELECT 可。フォロー機能で他ユーザー情報取得不可
- **R2 Codex**: 【維持】見落としの可能性が高い。公開列限定の参照ポリシーを計画に明記すべき
- **結論**: 採用 — Phase 0 として既存スキーマ修正を新設

#### A-6: ページネーション戦略
- **R1 Claude**: ページネーション方針の記載なし
- **R2 Codex**: 【修正】初期は LIMIT + ORDER BY で十分。ただし戦略未記載は避ける。`(created_at DESC, id DESC)` 複合インデックスを早期に作成すべき
- **結論**: 採用（修正版）— 全体ルールにページネーション方針を追記、Phase 3/4 にインデックスを追加

## 所見

1. **profiles RLS の見落とし（A-1）**: フォロー機能の前提条件として致命的。Claude が発見し Codex も合意。Phase 0 として追加
