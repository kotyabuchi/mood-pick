## 議論サマリー

### 合意点（初回で一致、即採用）

- C-1/A-1: テスト前提データの手順定義が必要
- C-3: ロール切替のリセット手順を明記
- C-4: WITH CHECK 系の不正操作テスト追加
- C-6/A-4: PROGRESS.md 更新タイミングの明記

### ラリーの記録

#### C-2: pgTAP/CI 自動化
- **R1 Codex**: SQL Editor 手動実行のみでは回帰検知不能。pgTAP を supabase/tests に追加し CI で自動実行すべき
- **R1 Claude**: PROGRESS.md に「将来的に CI 化を検討」と明記。Phase 6 のスコープは手動スモークテスト。pgTAP 導入は過剰
- **R2 Codex**: 【修正】スコープ制約は妥当。手動実施の再現性確保（setup/teardown と検証 SQL の保存）を必須、CI化は次フェーズ明記に修正
- **結論**: 修正して採用（CI 化はスコープ外、テスト SQL の保存で再現性確保）

#### C-5: /codex-debate の任意化
- **R1 Codex**: /codex-debate は環境依存で再現性が低く、Batch 完了ゲートに置くと実行不能になりうる
- **R1 Claude**: CLAUDE.md の Quality Checks に「必ず実行。スキップ禁止」と明記。プロジェクトルール変更不可
- **R2 Codex**: 【撤回】CLAUDE.md の必須ルールと矛盾するため撤回
- **結論**: 却下（プロジェクトルールにより変更不可）

#### A-2: avatars bucket の RLS テスト漏れ
- **R1 Claude**: storage.objects (avatars) の RLS テストがテーブル一覧から漏れている
- **R2 Codex**: 【維持】Supabase では storage.objects の RLS が実データ保護の本体。未検証は重要な考慮漏れ
- **結論**: 採用（テーブル一覧に avatars bucket を追加）

#### A-3: SECURITY DEFINER トリガー動作確認
- **R1 Claude**: トリガーの動作確認がスコープに含まれていない
- **R2 Codex**: 【修正】全ロジック試験は重いが、起動確認と権限境界の破綻がないことは必要。各1ケースの軽量テスト
- **結論**: 修正して採用（軽量テスト3件追加）
